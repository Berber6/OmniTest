## 三、项目开发

### 1. 项目意义和目标

软件测试中，测试用例的编写和执行是两块最大的体力活。文档站点有大量的功能描述，但要把它们翻译成可执行的测试脚本，再人工跑一遍，耗时且容易遗漏。本项目想解决两件事：

- **从文档到测试场景的自动生成**：把 4gaboards 文档站的页面爬下来，用 RAG + LLM 提取功能特征，生成结构化的测试场景（步骤 + 预期结果）
- **从场景到实际执行的自动化**：用 LLM Agent 驱动浏览器（Playwright MCP），根据场景步骤自主操作页面，验证实际结果是否符合预期，再做变异测试看执行框架能不能发现被篡改的步骤

目标不是替代人工测试，而是把"读文档 → 写用例 → 跑用例"这条链路尽可能自动化，让人聚焦在评审和边界设计上。

#### 关键算法与设计

**Task 1：RAG 管线**

```
crawl4ai 爬取 → RecursiveCharacterTextSplitter 分块 → bge-m3 嵌入 → ChromaDB 存储
                                                                      ↓
                                        LLM 提取功能特征 ← RAG 检索相关 chunks
                                                ↓
                                        LLM 生成结构化测试场景（步骤 + 预期）
```

特征提取时用 `validate_granularity` 校验功能粒度，过粗或过细的特征会被合并或拆分。场景生成时把特征的 source_chunks 作为上下文，保证场景与文档的对应关系。

**Task 2：Agent 执行循环**

每个场景独立起一个子进程，子进程内维持一个 Playwright MCP 会话和一个事件循环。流程：

1. 确定性登录（CSS 选择器直接操作，不让 LLM 猜登录）
2. 决策循环（最多 24 步）：
   - 取 Playwright accessibility snapshot
   - 拼 prompt（snapshot + 已执行步骤 + 场景步骤）送给 LLM
   - LLM 返回单步动作 JSON（click/type/navigate/select/hover/press_key/wait/drag）
   - 调 MCP 执行，每步自动截图
   - LLM 返回 `done=true` 时退出循环
3. 双策略验证：
   - 文本验证：把最终页面 snapshot 和场景预期送给 LLM，判断是否符合
   - 视觉验证：用 LLM 比对执行截图与参考图（爬取阶段保存的）
4. 反思重试（最多 1 次）：验证失败时让 LLM 反思原因，重新跑一遍

**变异测试**

对每个场景生成 3 类变异体（最多 5 个/场景）：

- 操作变异：改 click 目标（点错按钮）
- 输入变异：改输入值（无效字符、边界值）
- 步骤变异：删/重复/乱序步骤

变异体跑完后，记录 Agent 是否检测到错误以及错误类型。检测率反映了执行框架的鲁棒性。

### 3. 项目具体设计

#### BDD User Stories

以 4ga Boards 看板应用为例：

```
Feature: 看板项目管理
  Scenario: 创建新项目
    Given 用户已登录 demo.4gaboards.com
    When 用户点击"创建项目"按钮
    And 输入项目名称"测试项目"
    And 点击"确认"
    Then 看板列表中出现名为"测试项目"的项目

  Scenario: 拖拽卡片到另一列
    Given 看板中存在待办卡片
    When 用户把卡片从"待办"列拖到"进行中"列
    Then 卡片出现在"进行中"列
    And "待办"列中不再有该卡片
```

这些场景由 Task 1 自动生成，Task 2 执行时 LLM 会把 Given/When/Then 翻译成具体的浏览器操作。

#### Low-fi UI

前端主要页面：

- `/login` — 登录页（用户名/密码表单）
- `/` — Dashboard（统计卡片 + 管线状态）
- `/features` — 功能特征列表（分类筛选 + 搜索）
- `/features/[id]` — 特征详情（关联场景列表）
- `/scenarios` — 场景列表
- `/executions` — 执行记录列表（状态筛选）
- `/executions/[id]` — 执行详情（时间线 + 每步截图 + 验证结果）
- `/mutations` — 变异测试结果
- `/graph` — 依赖图与覆盖率
- `/settings` — 系统设置（模型、Token 用量、导入导出）

#### TDD Test Cases

后端关键测试：

- Task 1：爬取后 ChromaDB chunk 数 > 0；特征提取后 Feature 表非空；场景生成后 TestScenario.steps_json 可解析
- Task 2：Agent 循环在 MAX_LOOP_STEPS 内终止；done=true 时 executed_steps 非空；验证返回 passed/confidence/reason 三字段
- 变异测试：3 类变异体各生成至少 1 个；MutationResult 记录 detected_error_type

### 4. 创新点与特色功能

#### 用 MCP 协议而不是直接调 Playwright API

好处是 Agent 和浏览器之间多了一层标准化接口，未来换浏览器实现（比如换到 Puppeteer）只改 MCP server 不改 Agent 代码。坏处是多一层进程通信，调试比直接调 API 麻烦一些。

#### Accessibility snapshot 代替截图做决策

LLM 拿到的不是页面截图，而是 Playwright 的 accessibility tree（带 `[ref=eN]` 引用标记）。LLM 在 snapshot 里看到 `<button ref=e10>创建项目</button>`，下一步直接说 `click(e10)`，不需要先识别按钮在屏幕坐标。这比纯视觉方案（让 LLM 看截图点坐标）稳定得多。

#### 双策略验证

文本验证看页面 DOM 文本是否符合预期，视觉验证看截图是否与参考图一致。两者都过才算通过，避免单一策略的误判（比如页面有"创建成功"文本但弹窗实际没出来）。

#### 变异测试评估执行框架

这是项目比较有特色的一块。传统的测试用例评估关注覆盖率，我们关注的是执行框架能不能发现"被篡改的步骤"。如果变异后 Agent 仍然报告通过，说明框架的验证逻辑有问题。实测中操作变异和步骤变异的检测率较高，输入变异的检测率较低（因为很多输入框对无效字符是宽容的）。

### 5. 不足与改进

#### Agent 决策受 LLM 能力限制

DeepSeek-V4-Flash 在简单场景下表现可以，但遇到需要多步推理的场景（比如"先创建项目再创建卡片再拖拽"）时，会在中间步骤丢失上下文。改用更强的模型（比如 GLM-5.1 或 DeepSeek-V4）会改善，但成本和延迟会上升。

#### 0 步执行的兜底验证

早期版本中，如果 LLM 第一步就超时，Agent 会带着 0 个 executed_steps 进入验证，fallback 验证会因为"没有发现错误"而误判通过。后来加了显式的 0 步检查，但这种边界情况说明错误处理还需要更细致。

#### 截图与执行计划显示

开发过程中踩了几个坑：

- 后端 `serve_screenshot` 路由漏写 `return FileResponse`，所有截图请求返回 4 字节 "null"
- Next.js dev server 对带斜杠的 URL 做 308 重定向，浏览器在重定向时丢掉 Authorization header，导致 `/api/settings/` 被识别为 401 跳登录页
- 执行计划在前端展示时需要把场景步骤翻译成 Action 格式，早期版本直接返回空数组

这些都是具体且小的 bug，但确实花了不少时间定位。

#### RAG 覆盖率

Task 1 的场景生成依赖爬取到的文档页面。4gaboards 文档站有些功能（比如权限管理、Webhook）的文档页面层级较深，crawl4ai 的广度优先策略有时漏掉。改成深度优先 + sitemap 引导后有所改善，但仍有遗漏。

#### 改进方向

- Agent 决策换成更强的模型，或做模型路由（简单步骤用 Flash，复杂步骤用更强的模型）
- 变异测试增加语义变异（比如把"创建项目"改成"创建项目并立即删除"）
- Task 1 加入 sitemap.xml 解析，提高爬取覆盖率
- 前端加执行回放功能（把每步截图串成视频）

## 四、使用手册

### 1. 环境准备

#### 后端

```bash
cd backend
pip install -r requirements.txt
cp .env.example .env  # 编辑 .env 配置 LLM API、管理员账号等
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

`.env` 关键配置：

```
LLM_API_BASE_URL=https://your-llm-endpoint/v1
LLM_API_KEY=your-key
ADMIN_USERNAME=admin
ADMIN_PASSWORD=admin
LLM_MODEL_GENERATION=deepseek-v4-flash
```

#### 前端

```bash
cd frontend
npm install
npm run dev  # 默认 :3000
```

前端通过 `next.config.ts` 的 rewrites 把 `/api/*` 和 `/ws/*` 代理到后端 :8000。

#### Docker 一键启动

```bash
docker-compose up -d
```

### 2. 登录

> **系统截图占位符**：登录页面截图
> `![登录页](screenshots/login.png)`

浏览器打开 `http://localhost:3000`，自动跳转到 `/login`。输入管理员账号密码（默认 admin/admin，正式部署请改 `.env`），点击登录。

### 3. Task 1：文档爬取与场景生成

#### Step 1：爬取文档

进入 Dashboard，点击"爬取文档"按钮，或访问 Task 1 管线页面触发爬取。

> **系统截图占位符**：爬取进度截图
> `![爬取进度](screenshots/crawl.png)`

爬取进度通过 `/api/task1/crawl/status` 轮询，状态包含已爬取页面数、当前 URL、是否完成。

#### Step 2：提取功能特征

爬取完成后点击"提取特征"。LLM 会从 ChromaDB 的 chunks 中提取功能特征，存入 `features` 表。

> **系统截图占位符**：功能特征列表
> `![特征列表](screenshots/features.png)`

在 `/features` 页面可以按分类筛选、搜索特征。点击特征进入详情页，查看关联的场景。

#### Step 3：生成测试场景

选择若干特征，点击"生成场景"。LLM 为每个特征生成结构化测试场景（步骤 + 预期结果）。

> **系统截图占位符**：场景列表
> `![场景列表](screenshots/scenarios.png)`

### 4. Task 2：场景执行

#### 单场景执行

在 `/scenarios` 页面，点击场景右侧的"执行"按钮。后端会起一个子进程跑 Agent，执行进度通过 WebSocket 实时推送到前端。

> **系统截图占位符**：执行详情页
> `![执行详情](screenshots/execution.png)`

执行详情页（`/executions/[id]`）展示：

- 执行计划（场景步骤翻译成 Action 格式）
- 时间线（每步的动作、目标、结果、截图）
- 验证结果（passed/failed、confidence、reason）
- 反思记录（如果触发了重试）

#### 取消执行

执行过程中点击"取消"按钮，后端会终止子进程，状态变为 `cancelled`。

### 5. 变异测试

在 `/scenarios` 页面，点击场景右侧的"变异测试"按钮。后端为该场景生成 3 类变异体（最多 5 个），逐个执行。

> **系统截图占位符**：变异测试结果
> `![变异结果](screenshots/mutations.png)`

`/mutations` 页面展示所有变异结果，可以按变异类型、检测到的错误类型筛选。每个变异记录包含：

- 变异类型（操作/输入/步骤）
- 变异描述
- 执行状态
- 检测到的错误类型（none 表示未检测到，其他值表示检测到）
- 关联的执行记录

### 6. 依赖图与覆盖率

> **系统截图占位符**：依赖图
> `![依赖图](screenshots/graph.png)`

`/graph` 页面用 React Flow 可视化 Feature → Scenario → Execution → Mutation 的依赖关系。需要 Neo4j 支持（默认关闭，在 `.env` 中开启）。

### 7. 系统设置

> **系统截图占位符**：设置页
> `![设置页](screenshots/settings.png)`

`/settings` 页面：

- 动态配置（LLM 模型、API 地址等，修改后立即生效）
- 模型选择（生成/推理/视觉三个角色，可分别配置模型和 fallback 链）
- Token 用量统计（按模型、阶段汇总，含成本估算）
- Token 用量明细（分页查询每条调用记录）

### 8. 导入导出

`/settings` 页面底部提供导入导出功能：

- 导出：features / scenarios / executions / bundle（全部）
- 导入：上传 JSON 文件，系统会去重并导入

### 9. Dashboard

> **系统截图占位符**：Dashboard
> `![Dashboard](screenshots/dashboard.png)`

首页 Dashboard 展示：

- 特征数、场景数、执行数、成功率
- 变异数、变异检测率
- 爬取页面数、ChromaDB chunk 数
- Task 1 管线状态（爬取/提取/生成）
