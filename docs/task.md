# 基于大模型的测试场景生成与智能测试工具 — 任务要求详解

## 1. 项目简介

随着 Web Agent 快速发展，人们对其在复杂 Web 应用中的测试可用性提出了更高要求。现有 Web Agent 在 WebArena 等基准上仍表现出较高的任务失败率，原因在于：

1. Web 应用业务逻辑复杂且高度领域化
2. 大模型存在幻觉，且缺乏待测应用的领域知识，容易输出与事实不符的信息
3. 难以在执行完操作后准确判定任务是否成功

软件文档（如用户手册）蕴含大量业务知识，可帮助 Agent 更好地理解功能、判断执行结果。本课程实践旨在设计一种基于大模型的测试场景生成与自主执行工具。

## 3. 任务一：基于用户手册的测试场景自动生成

### 目标

基于大模型检索增强技术（RAG），从用户手册中提取知识，生成测试场景。

### 具体要求

1. **功能点提取**: 能够根据用户手册提取软件的主要功能点
2. **结构化测试场景生成**: 根据每个功能点能够生成结构化的测试场景
3. **可视化展示**: 支持功能点、测试场景的可视化展示

### 测试场景构成格式

测试场景结构: `[ [step]+ [expectation]? ]+`

即：一个测试场景由**一个或多个步骤-预期状态对**组成，具体包含：

- **操作步骤 (step)**: 完成功能所需的具体操作步骤（必填，可有多步）
- **预期状态 (expectation)**: 功能完成后的预期状态，相当于测试预言，包含用于评估功能是否成功完成的关键特征（可选）

---

## 4. 任务二：测试场景驱动的智能测试智能体

### 目标

构建基于大模型的 Web 测试智能体，自动执行任务一生成的测试场景。

### 架构要求（至少包含以下要素，可自主扩展）

1. **规划 (Planning)**: 根据测试场景自主规划执行计划
2. **记忆 (Memory)**: 辅助执行的上下文信息
3. **执行 (Execution)**: 与浏览器交互，执行规划动作
4. **验证 (Verification)**: 根据整个测试轨迹验证测试场景是否执行成功，该功能是否正确

---

## 5. 评分标准

### 5.1 基础功能档

**任务一**:
- 根据用户手册能够识别主要功能点
- 生成主要功能的测试场景，测试场景的生成格式符合要求

**任务二**:
- 智能体能够执行任务一中识别的简单测试场景
- 能够验证执行完整性与功能正确性

### 5.2 提升创新档

**任务一**:
- **正确性与全面性**: 确保从用户手册等文档中检索到的信息真实有效，尽量避免大模型"幻觉"导致的错误；尽可能全面地覆盖文档中包含的功能并避免遗漏
- **粒度与可执行性**: 功能划分的粒度应合适，既不能过细导致难以组合成实际功能，也不宜过粗以致无法指导具体操作；同时保证生成的测试场景具有可执行性，即能够被现有的 Web 测试智能体有效执行

**任务二**:
- **通过率与稳定性**: 提升智能体执行测试场景的准确率，对于简单、中等难度测试场景能够正确通过，甚至通过部分困难场景；也可考虑提升智能体执行的效率
- **变异测试**: 支持对测试场景进行变异，能够识别典型的应用错误（包括：执行异常、布局问题、语义错误等）
- **验证能力**: 能够根据执行轨迹正确识别执行结果（成功与失败），对于执行失败准确输出失败原因
- **鼓励**: 所有创新方法设计以及评估策略设计（评估方法与指标）等

---

## 6. 补充要求

- **计算设备**: 对个人计算设备的计算能力无特殊要求，能完成赛题即可
- **软件依赖**: 编程语言及所用技术不限；需使用版本控制工具（Git 等）
- **大模型**: 国产大模型 GLM-4.6V/4.7、Qwen3-VL-Plus、DeepSeek V3

## 7. 任务分解与关键技术点

### 任务一关键技术

| 技术点 | 说明 |
|--------|------|
| RAG (检索增强生成) | 从用户手册中检索相关知识，减少大模型幻觉 |
| 文档解析与索引 | 解析 4gaboards 用户手册文档，建立知识库索引 |
| 功能点提取 | 从文档中识别并提取主要功能模块 |
| 测试场景生成 | 按格式 `[step+ expectation?]` 生成结构化测试场景 |
| 可视化展示 | 提供功能点和测试场景的图形化/交互式展示界面 |
| 粒度控制 | 功能划分粒度适中，保证可执行性 |

### 任务二关键技术

| 技术点 | 说明 |
|--------|------|
| 规划模块 | 根据测试场景制定浏览器操作执行计划 |
| 记忆模块 | 维护执行上下文（页面状态、已执行步骤、历史信息） |
| 执行模块 | 与浏览器交互（点击、输入、导航等），执行规划动作 |
| 验证模块 | 对比预期状态与实际状态，判定功能是否正确 |
| 变异测试 | 对测试场景变异，检测执行异常、布局问题、语义错误等 |
| 失败原因分析 | 执行失败时输出准确的失败原因 |
